# Docker Maintenance Operations

This runbook covers the home-network Docker maintenance control plane.

## Purpose

The maintenance workflow answers three questions for each Docker host:

- Can disk space be safely reclaimed?
- Are Docker images or source-built services likely to need updates?
- Is the host configured for report-only, manual approval, or automatic action?

Phase 1 is read-only. It does not prune Docker data, pull images, recreate containers, or update code.

## Host policy

Policy lives in `inventory/hosts.yml` under `hosts.<host>.maintenance`.

Modes:

- `report_only`: collect facts and show proposals only.
- `manual_approval`: collect facts and propose exact commands; an operator must run an action explicitly.
- `automatic`: execute only allowlisted low-risk actions during the configured window.

Initial policy:

- `jellybase`: `manual_approval`, because `/` has recently fired `FilesystemSpaceLow`.
- `jellyhome`: `report_only`.
- `jellyberry`: `report_only`.
- `jellybackup`: `report_only`.
- `jellysa`: `report_only`.

Safety defaults:

- Volume pruning is disabled everywhere.
- Broad `docker system prune -a` is not generated by default.
- Image and code updates are report-only in Phase 1.
- Automatic mode is not enabled for any host initially.

## Read-only collector

Run locally from the repo:

```bash
python scripts/docker-maintenance-check --host jellybase --check-only --pretty --stdout
```

Write a JSON report:

```bash
python scripts/docker-maintenance-check \
  --host jellybase \
  --check-only \
  --pretty \
  --output /tmp/docker-maintenance-jellybase.json
```

Write Prometheus textfile metrics:

```bash
python scripts/docker-maintenance-check \
  --host jellybase \
  --check-only \
  --output /tmp/docker-maintenance-jellybase.json \
  --metrics-output /var/lib/node_exporter/textfile_collector/docker_maintenance.prom
```

Use sudo only if the textfile collector directory requires it. The collector itself is read-only.

## Report contents

The JSON report includes:

- `root`: filesystem usage for `/`.
- `docker.system_df`: parsed `docker system df` output.
- `docker.safe_reclaim_gb`: low-risk reclaim estimate.
- `docker.unused_images`: a bounded list of unused image rows for review.
- `proposed_actions`: exact cleanup commands and policy gates.
- `updates`: placeholder status for image/code updates; Phase 1 does not implement update detection.

## First jellybase cleanup path

Current evidence showed jellybase root at about 90% used. The safe first action is old BuildKit cache pruning, if the generated report proposes it:

```bash
python scripts/docker-maintenance-check --host jellybase --check-only --pretty --output /tmp/docker-maintenance-jellybase.json
python -m json.tool /tmp/docker-maintenance-jellybase.json | less
```

If the plan proposes `jellybase-builder-prune-168h`, the manual command is:

```bash
docker builder prune --filter until=168h --force
```

Then verify:

```bash
df -hT /
docker system df
python scripts/docker-maintenance-check --host jellybase --check-only --pretty --output /tmp/docker-maintenance-jellybase-after.json
```

Do not run `docker system prune -a` as the first action.

## Systemd timer install

Install the read-only timer on a Docker host:

```bash
scripts/install-docker-maintenance-check --start-now
```

This installs:

- `home-network-docker-maintenance-check.service`
- `home-network-docker-maintenance-check.timer`

The service runs `scripts/docker-maintenance-run`, which writes:

- JSON report: `/opt/docker/appdata/docker-maintenance/<host>.json`
- metrics: `/var/lib/node_exporter/textfile_collector/docker_maintenance.prom`

If the collector fails, the wrapper writes failure metrics instead of silently leaving stale success metrics.
