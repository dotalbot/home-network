# Obsidian on jellybase

## Purpose

`obsidian-jellybase` provides a browser-accessible LinuxServer Obsidian desktop for trusted LAN/Tailscale use.

Image:

- `lscr.io/linuxserver/obsidian:latest`

Access:

- LAN: `https://192.168.1.2:30302`
- Tailscale: `https://100.125.86.118:30302`

The image uses a self-signed HTTPS certificate, so a browser certificate warning is expected unless the service is later put behind a trusted reverse proxy.

## Security boundary

This is a desktop GUI container. Do not expose it to the public internet.

Current posture:

- bound only to jellybase LAN and Tailscale IPs;
- basic HTTP auth via `CUSTOM_USER` plus `FILE__PASSWORD`;
- password stored outside Git at `/opt/docker/.secrets/obsidian-jellybase/password`;
- no Docker socket mount;
- no privileged mode;
- no GPU passthrough in the first deployment.

## Vault location

The user-facing vault is under the home-network Docker appdata tree, but it is explicitly writable by the `jellypai` host user:

- Host path: `/opt/docker/appdata/obsidian-jellybase/vault`
- Container path: `/config/vault`
- Runtime ACL: `jellypai:rwx`
- `jellypai` UID/GID: `1005:1005`

The container runs with:

- `PUID=1005`
- `PGID=1005`

This makes files created through the Obsidian GUI owned by `jellypai` on the host. The runtime directory also has an explicit ACL so the `jellypai` account can read and write the vault directly.

Persistent container settings live separately at:

- `/opt/docker/appdata/obsidian-jellybase/config`

## Source files

- `docker/hosts/jellybase.yaml`
- `inventory/services.yml`
- `inventory/backups.yml`
- `docker/appdata/homepage/services.yaml` generated by `scripts/homepage-render`

## Runtime bootstrap

Run on jellybase before first deploy:

```bash
sudo install -d -m 0750 -o root -g dockerops /opt/docker/.secrets/obsidian-jellybase
sudo sh -c 'test -s /opt/docker/.secrets/obsidian-jellybase/password || openssl rand -base64 24 > /opt/docker/.secrets/obsidian-jellybase/password'
sudo chown root:dockerops /opt/docker/.secrets/obsidian-jellybase/password
sudo chmod 0640 /opt/docker/.secrets/obsidian-jellybase/password
sudo install -d -m 0775 -o 1005 -g 1005 /opt/docker/appdata/obsidian-jellybase/config
sudo install -d -m 0775 -o 1005 -g 1005 /opt/docker/appdata/obsidian-jellybase/vault
sudo setfacl -m u:jellypai:rwx -m d:u:jellypai:rwx /opt/docker/appdata/obsidian-jellybase/vault
```

Deploy from the jellybase runtime checkout:

```bash
cd /home/jellyfish/repo/home-network/docker
just sync-docker-config
just up obsidian-jellybase
```

If the container-create settings change, recreate with `just up obsidian-jellybase`; do not rely on `docker compose restart`.

## Verification

```bash
cd /opt/docker
docker compose --env-file .env -f docker-compose.yml -f hosts/jellybase.yaml config >/tmp/obsidian-compose.yml
docker inspect obsidian-jellybase --format 'status={{.State.Status}} image={{.Config.Image}}'
docker inspect obsidian-jellybase --format '{{range .Mounts}}{{println .Source "->" .Destination}}{{end}}'
namei -l /opt/docker/appdata/obsidian-jellybase/vault
getfacl /opt/docker/appdata/obsidian-jellybase/vault
sudo -u jellypai sh -c 'touch /opt/docker/appdata/obsidian-jellybase/vault/.jellypai-write-test && rm /opt/docker/appdata/obsidian-jellybase/vault/.jellypai-write-test'
curl -k -s -o /dev/null -w '%{http_code}\n' https://192.168.1.2:30302/
```

Expected HTTP result without credentials is `401` or a login page response from the LinuxServer desktop stack. Authenticated browser access should open Obsidian; select `/config/vault` as the vault folder.

## Backup notes

`/opt/docker` remains covered by the jellybase Docker-root backup set. The Obsidian vault is inside `/opt/docker/appdata/obsidian-jellybase/vault`, so it is covered by that host-level Docker backup path.
