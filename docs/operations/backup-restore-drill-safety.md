# Backup Restore Drill Safety Guide

Status: phase 4 design; documentation-only
Last updated: 2026-06-01

## Purpose

This guide defines the safe default for future `home-network` restore drill automation. It is intentionally scratch-only: it describes how automation should prove that Borg archives and database dumps are usable without writing to production paths, stopping services, changing privileged host configuration, or restoring live data.

Phase 4 does not run a production restore and does not mutate privileged host state. Future tooling should generate reviewed commands and manifests first, then rely on explicit operator approval before anything touches live services.

## Safety invariants

A restore drill tool must:

1. Extract only into a new timestamped scratch directory, normally under `/tmp/home-network-restore-drill/`, created with mode `0700` or stricter because restored appdata/database dumps may contain sensitive data.
2. Refuse production-looking destinations by default.
3. Canonicalize paths with `realpath`/path ancestry checks and reject symlink escapes; do not rely on string-prefix checks alone.
4. Require a service/runbook/backup-set selection from `inventory/backups.yml`; no arbitrary path restore UI.
5. Keep Borg passphrases, SSH private keys, exported repo keys, database passwords, `.env` contents, and raw secret files out of logs, docs, Discord, and future UI payloads.
6. Validate restored data in scratch with service-specific checks.
7. Emit only secret-free drill evidence: host, service, archive, destination label, paths restored, validators run, pass/fail, and cleanup status.
8. Stop after a scratch drill. Production restore is a separate approval-gated maintenance workflow.

## Explicitly forbidden default destinations

The tool must reject these destinations unless a future production-restore mode is added with a separate approval gate:

- `/`
- `/opt`
- `/opt/docker`
- `/opt/docker/appdata`
- any live `/opt/docker/appdata/<service>` path
- any live PostgreSQL/MySQL/MariaDB data directory
- any live SQLite database path
- `/home/jellyfish/media`
- existing bind-mount roots used by Docker services
- any path that already exists and is not inside the tool-created scratch root

A safe default scratch path shape is:

```text
/tmp/home-network-restore-drill/<host>/<service>/<archive-or-latest>/<timestamp>/
```

The implementation should create that directory with a restrictive umask, record the exact path in the result manifest, and clean up only that exact directory after operator review.

## Restore drill manifest

Future automation should generate a reviewable manifest before extraction. Minimum fields:

```yaml
version: 1
mode: scratch_drill
host: jellybase
service: central-postgres
destination_label: primary
archive: latest
backup_set_id: central-postgres-logical-dumps
restore_runbook: docs/runbooks/central-postgres-manyfold-restore.md
scratch_root: /tmp/home-network-restore-drill/jellybase/central-postgres/latest/20260601T030000Z
paths:
  - opt/docker/appdata/postgres/logical-dumps/latest
validators:
  - type: postgres_logical_dump_restore
    temporary_container: true
    network: none_or_local_only
approval:
  required_for_scratch: true
  required_for_production: always_separate_gate
```

Manifest rules:

- Paths are archive-relative or known safe source paths from inventory/runbooks.
- The scratch root is generated by the tool, not typed free-form in a web UI.
- `mode` defaults to `scratch_drill`; production mode is not accepted by the phase 4 design.
- Approval records must not include passwords or tokens.

## Operator approval gates

Use these gates before automation progresses:

### Gate A — Plan review

Before running any command, the operator reviews:

- selected host;
- selected service/backup set;
- Borg destination label;
- archive selection policy or exact archive name;
- archive-relative paths to extract;
- scratch root;
- validators;
- cleanup plan.

The future UI/API may generate the manifest, but it should not run the drill until the operator explicitly approves this reviewed plan.

### Gate B — Scratch extraction approval

A scratch extraction approval allows only:

- creating the scratch directory;
- running `borgmatic extract`/equivalent against selected paths;
- running validators inside scratch or temporary isolated containers;
- writing a secret-free drill result.

It does not allow:

- stopping services;
- writing to `/opt/docker`;
- replacing databases;
- changing systemd units;
- editing `/etc/borgmatic/config.yaml`;
- enabling/disabling timers.

### Gate C — Production restore escalation

If a drill shows that a production restore is needed, stop and create/execute a service-specific production restore runbook. Minimum separate approval requirements:

1. maintenance window and user impact confirmed;
2. latest backup/archive selected;
3. pre-restore local snapshot or dump planned;
4. secrets available from password manager or host-local secret store;
5. exact services to stop and restart listed;
6. rollback artifact retention agreed;
7. health checks and acceptance criteria listed.

Production restore is outside the restore-drill automation scope in this phase.

## tmux and sudo handoff model

Some hosts require interactive sudo or are already open in operator-managed tmux panes. Future automation should support a handoff mode instead of trying to hide privileged work.

Recommended pattern:

1. Generate the scratch-drill manifest and a command block.
2. If a tmux pane is supplied, capture it first to confirm hostname and working directory.
3. Send one command at a time to the named pane only after operator approval.
4. If sudo prompts, stop and ask the operator to type the password directly in tmux. Do not ask for passwords in chat, commit logs, or UI fields.
5. Capture output after each stage and redact or reject secret-bearing output before publishing a status.
6. Record the final manifest/result path and cleanup status.

Suggested command staging:

```text
stage 1: print hostname, user, date, and selected manifest path
stage 2: create scratch root
stage 3: run Borg extraction into scratch
stage 4: run validators
stage 5: print secret-free result manifest
stage 6: optional cleanup after operator confirms evidence is captured
```

## Validators by data type

### Generic files/appdata

- Confirm expected files/directories exist under scratch.
- Confirm file sizes are plausible where runbooks specify minimums.
- Optionally compare known sentinel/config files.
- Do not recursively list private appdata in public logs.

### SQLite

- Restore the `.sqlite`, `.db`, or service-specific file plus `-wal` and `-shm` sidecars when present.
- Run `sqlite3 <scratch-db> 'PRAGMA integrity_check;'`.
- Run service-specific read-only queries only when documented in the runbook.
- For production restore planning, require service stop or checkpoint procedure before replacing live files.

### PostgreSQL

- Prefer logical dumps from `/opt/docker/appdata/postgres/logical-dumps`.
- Restore into a temporary PostgreSQL container or scratch database, not the live `central-postgres` container.
- Validate globals and required databases as described in the service runbook.
- Keep database passwords out of logs; use temporary local trust/socket or mounted secret files only if needed and never display values.

## Result record

A completed scratch drill should record:

```yaml
status: pass
host: jellybase
service: central-postgres
archive: jellybase-2026-06-01T03:00:00
backup_set_id: central-postgres-logical-dumps
scratch_root: /tmp/home-network-restore-drill/jellybase/central-postgres/latest/20260601T040000Z
validators:
  - name: pg_restore_manyfold_scratch
    status: pass
cleanup:
  status: complete
  removed_path: /tmp/home-network-restore-drill/jellybase/central-postgres/latest/20260601T040000Z
notes: no production paths modified
```

If cleanup is deferred, record that explicitly and include the exact path that remains.

## Implementation notes for future `scripts/backup-restore-drill`

The first implementation should be conservative:

- support `--dry-run`/manifest generation first;
- require `--approve-scratch` for extraction;
- require host and backup set IDs from `inventory/backups.yml`;
- default archive to latest only after confirming latest-success status;
- reject production paths with a hardcoded denylist plus canonicalized path ancestry checks that reject symlink escapes;
- run a disk-space preflight before extraction, especially for database dumps and large appdata paths;
- write result JSON atomically;
- avoid shelling out with unquoted user input;
- include `bash -n` and manifest fixture tests before rollout.

No production restore flag should be added until the operator guide and service runbooks define a separate approval path.
